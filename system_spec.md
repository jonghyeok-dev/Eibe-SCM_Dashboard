## CH 1. 시스템 아키텍처 및 구동 제약 조건

### 1.1. 인프라 구동 가이드

1. 본 시스템은 클라우드 인프라 비용 및 데이터 유출을 원천 차단하기 위해 사내 호스트 PC의 로컬 자원(8GB RAM)만을 사용하는 **독립형 로컬 웹 어플리케이션**이다.
2. 데이터베이스는 **SQLite 3** 파일 기반 경량 DB를 사용한다. (`data/local_erp.db`)
3. 백엔드는 **FastAPI + Uvicorn** 조합으로 로컬 포트 서빙한다.
4. 프론트엔드는 백엔드가 정적 파일로 서빙하며, 사내망 내 타 PC 접근을 위해 포트를 개방한다.

### 1.2. 절대 금지 조항

1. **머신러닝 라이브러리 도입 금지** — 소량 데이터 과적합 방지. 모든 예측은 사칙연산 기반 통계 평탄화로 제한한다.
2. **변동 단가 추적 배제** — 모든 제품 외화 매입 단가는 연간 고정 계약 구조이므로, 상품 마스터의 고정 단가만 사용한다.
3. **클라우드 인증 배제** — 자체 SQLite 기반 계정 체계 사용 (ADMIN / OPERATOR 역할, Bcrypt 해싱).
4. **접근 제어 방식 (개발 후반부 적용 예정)** — 로컬 및 지정된 사내 IP에서만 접근 가능하도록 IP 화이트리스트 기반 API 인증 로직을 추가할 예정 (현재 여러 PC에서 Git 기반 분산 개발 중이므로, IP 차단 로직은 전체 플로우 구현 및 테스트 완료 후 최후반부에 활성화함).

### 1.3. 윈도우 무인 구동

- `start_server.bat`를 윈도우 작업 스케줄러에 등록하여 시스템 시작 시 자동 구동한다.
- Uvicorn: `uvicorn app.main:app --host 0.0.0.0 --port 8000`

## CH 3. 데이터베이스 개요

모든 테이블은 **3차 정규화(3NF)** 규칙에 의거하여 설계한다. 대리 기본키는 `INTEGER PRIMARY KEY AUTOINCREMENT`를 사용한다. 정확한 스키마는 `app/models.py`에서 관리한다.

### 3.0. 시스템 관리 영역
- **USER_ACCOUNT** — 사용자 계정 (username, role, password_hash)
- **SYSTEM_SNAPSHOT** — 3시간 단위 자동 백업 및 수동 백업 추적

### 3.1. 기준 정보 영역
- **PRODUCT_MASTER** — 상품 고유 식별, 연간 고정 단가, 입수량, MOQ
- **FFC_MASTER** — 풀필먼트 거점, 유통기한 임계일수, 이관 MOQ
- **FFC_PRODUCT_MOQ** — 창고-상품별 이관 MOQ 오버라이드
- **LOGISTICS_COST_MASTER** — 거점 간 카툰당 물류 단가

### 3.2. 입고 및 파이프라인 영역
- **INBOUND_LIST** — 코어 공급망 파이프라인 이력 (인보이스, B/L, 수량, 환율 등)
- **EXPECTED_INBOUND** — 해상 운송/통관 대기 물량
- **ORDER_QUANTITY** — 발주수량 (매월 발주한 상품코드별 수량)
- **PRODUCTION_COMPLETE** — 생산완료수량 (공급사 생산완료 raw 데이터)
- **INVOICE_QUANTITY** — 인보이스수량 (인보이스 발행 데이터)

### 3.3. 실적 및 계획 데이터 영역
- **CURRENT_INVENTORY** — 거점별 실시간 실재고 (입고 배치 연계)
- **OUTFLOW_HISTORY** — 주차별 재고 소멸 로그 (기초-기말 차이)
- **SALES_HISTORY** — 주차별 실제 판매량 데이터
- **TRANSFER_PLAN** — 거점 간 재고 이관 계획 및 물류비 예측
- **MATCHING_HISTORY_LOG** — 수동 매칭 학습 로그
- **MONTHLY_ORDER_PLAN** — 월별 발주 계획 (시스템 제안 + 실무자 수정)

### 3.4. 입고 관리 매칭 구조

입고 관리는 3단계로 분리하여 관리한다:

1. **발주수량 (ORDER_QUANTITY)** — 발주월에 상품코드별로 발주한 수량 (MONTHLY_ORDER_PLAN에서 파생)
2. **생산완료수량 (PRODUCTION_COMPLETE)** — 공급사 생산완료 raw 파일 업로드
3. **인보이스수량 (INVOICE_QUANTITY)** — 인보이스 발행 데이터 업로드

매칭 키: **상품코드 + 세일즈 오더 No. + 생산년월 넘버 + 인보이스 번호**

- 완벽 일치 항목은 자동 매칭 처리
- 미매칭 항목은 메인 요약 페이지에 "할일" 알림으로 노출하여 사용자 수기 매칭을 유도

## CH 4. 데이터 수집 및 멱등성 보장

### 4.1. 엑셀 기반 업로드 규칙

1. 모든 데이터는 시스템 제공 **고정 엑셀 템플릿**을 통해 업로드한다.
2. 업로드 파일은 `[YYYYMMDD_HHMMSS]_[실무자ID]_[템플릿유형].xlsx` 형식으로 `data/backups/`에 원본 보관한다.
3. 대시보드 화면 갱신은 실무자가 **새로고침** 버튼을 누를 때만 동작한다. 상단에 마지막 업데이트 일시를 표시한다.
4. 3시간마다 DB 스냅샷을 자동 생성하여 복구 안정성을 확보한다.

### 4.2. 파이프라인 매칭 시스템

- 발주 → 생산완료 → 인보이스 → 입고 단계를 모니터링한다.
- 자동 매칭: 상품코드 + 세일즈 오더 + 생산년월로 완벽 일치하는 건을 자동 매칭한다.
- 수동 매칭: 불일치 내역은 시스템이 자동 필터링하여 알림을 제공하고, 수정 가능한 UI를 통해 확정 처리한다.

## CH 5. SCM 코어 연산 엔진 요약

모든 연산은 `app/core/forecasting.py`에 사칙연산 기반으로 구현한다.

### 5.1. 주차별 출고량 평탄화

- 직전 12주 단순 출고량 평균 → 향후 24주 수요 예측 상수
- 단순 출고량 = 해당 주차 기초 재고 - 기말 재고

### 5.2. 동적 감모 버퍼

- 직전 12주간 (단순 출고량 - 순수 판매량)의 평균값
- 물류 손실(파손, 폐기 등)을 반영하여 안전재고에 가산

### 5.3. 미래 재고 시뮬레이션 (24주)

- 예상 기말재고(W) = 이전 주차 기말재고 + 입고량 - (출고 평균 × 가중치 + 감모 버퍼)
- 가중치: 프론트엔드 슬라이더로 조절 (기본 1.0)
- 6개월 뒤 안전재고 미만 시 발주 경고 및 MOQ 단위 규격화 제안

### 5.4. 비상 에어 수송 트리거

- 3개월 뒤(12주) 재고 마이너스 감지 시 항공 전환 팝업 제안

## CH 6. 풀필먼트 및 재고 자산 시뮬레이터

### 6.1. FEFO 유통기한 필터링

- 잔여 유통기한 < 거점 허용 기준 → 해당 수량을 가용 재고에서 제외 (락인 처리)
- 폐기 대상으로 격리 표기

### 6.2. 재고 자산 가치 산출

- 전사 총 현재고 금액 = Σ(캔수 × 고정단가 × 입고 당시 환율)
- 폐기 손실 금액 = 락인 수량 × 해당 배치의 실제 결제 환율 × 고정단가

### 6.3. 교차 이관 및 물류비 시뮬레이터

- 유통기한 임박 창고 → 기준 완화된 채널로 이관 추천
- 이관 수량은 MOQ 배수로 규격화
- 예상 물류비 = 이관 카툰 수 × 카툰당 물류 단가

## CH 7. 프론트엔드 메뉴 구조 및 UI 원칙

좌측 확장/축소 가능한 사이드바 레이아웃을 채택한다. 하단에 로그인 계정 정보 및 권한을 표기한다.

### 7.1. 6대 핵심 메뉴

**① 요약 (`/`)**
- 상단 KPI 카드: 총 현재고, 재고자산 금액, 품절 위험 품목, 폐기 리스크 총액
- 이중 트랙 시각화 차트 (수요 예측 + 재고 소멸 + 실제 판매량)
- 할일 알림 위젯: 미매칭 항목 수, 긴급 이관 필요 품목 등

**② 현재고 현황 (`/inventory`)**
- 거점별 현재고 테이블 상세
- 재고 이관 추천 및 MOQ 설정 위젯

**③ 유통기한 관리 (`/expiry`)**
- FEFO 기반 유통기한 임박 리스트
- 폐기 예상 금액 및 추가 판매량 도출 계산기

**④ 발주 계획 (`/order-plan`)**
- 대상 연월 기반 시스템 역산 제안 수량 확인
- 실무자 수정 저장 편집 그리드
- 히트맵 (재고일수) 및 시뮬레이션 차트

**⑤ 입고 관리 (`/matching`)**
- **3탭 구조**: 발주수량 / 생산완료수량 / 인보이스수량
- 각 탭에서 엑셀 업로드, 데이터 테이블, 매칭 상태 확인
- 자동 매칭 실행 및 수기 매칭 UI
- 고정 엑셀 양식 다운로드

**⑥ 시스템 운영 (`/users` — Admin 전용)**
- 사용자 계정 CRUD (ID, PW, 역할, 이름)
- 전체 DB 강제 스냅샷 백업
- 물류비 마스터 관리 (출발창고-도착창고 카툰당 단가)
- 이관 MOQ 관리

## CH 8. 보안 및 접근 제한

### 8.1. 인증 체계

1. 모든 데이터 변경 API(POST/PUT/DELETE)는 JWT Bearer 토큰 인증 필수.
2. 관리자 전용 API(사용자 관리, 스냅샷)는 `ADMIN` 역할 검증 추가.
3. 데이터 조회 API(GET)도 로그인 상태에서만 접근 가능하게 구현한다.
4. 비밀번호는 bcrypt로 해싱하여 저장한다.
5. JWT 토큰 유효시간: 12시간.

### 8.2. IP 화이트리스트 접근 제한 (마지막 단계 구현)

> ⚠️ 현재 Git 기반 멀티 PC 개발 환경이므로, **IP 제한은 서비스 배포 직전 마지막 단계**에서 구현한다.

1. **구현 방식**: FastAPI 미들웨어로 요청 IP를 검사하여, 허용된 IP 목록에 없으면 `403 Forbidden` 반환.
2. **화이트리스트 관리**: `data/allowed_ips.json` 파일에 허용 IP 목록을 관리한다.
3. **기본값**: `["127.0.0.1", "0.0.0.0"]` (로컬 접근 허용)
4. **관리 UI**: 시스템 운영 페이지에서 Admin이 IP를 추가/제거할 수 있는 위젯을 제공한다.
5. **개발 모드**: 환경변수 `SCM_DEV_MODE=1` 설정 시 IP 제한을 비활성화한다.
