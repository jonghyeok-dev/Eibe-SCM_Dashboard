# EIBE SCM Dashboard — System Spec & Project Rules

> **Last Updated:** 2026-06-21
> 이 문서는 본 프로젝트의 기획(System Specification)과 개발 지침(Project Rules)을 통합한 핵심 가이드라인 문서입니다. AI 에이전트와 개발자는 이 문서의 규칙을 최우선으로 준수해야 합니다.

## 1. 시스템 비전 및 개발 방향성

### 1.1. 배경 및 문제 의식
기존의 SCM(Supply Chain Management) 프로세스는 수많은 엑셀 파일을 이메일로 주고받으며 수기 매칭과 직관에 의존해왔습니다. 이는 데이터 파편화를 낳고, 악성 재고나 예기치 않은 품절(Out of Stock) 사태를 야기하며, 결국 폐기 비용 증가와 기회비용 손실로 이어집니다.

### 1.2. 프로젝트 목표 (Vision)
본 **EIBE SCM Dashboard**는 산재된 물류 파이프라인(발주-생산-인보이스-입고) 데이터를 하나의 뷰로 통합하고, 과거 판매/출고 데이터를 기반으로 한 **데이터 주도적 발주 예측(Data-driven Forecasting)**을 제공하는 것을 목표로 합니다.
실무자는 단순 반복 작업에서 해방되어, "이 재고를 언제 어디로 이관해야 물류비가 최적화될까?", "언제 발주를 넣어야 재고 부족을 막을 수 있을까?" 와 같은 **전략적 의사 결정**에 집중하게 됩니다.

### 1.3. 수요 예측 및 발주 시뮬레이션 철학
복잡한 딥러닝 기반 예측보다는, 실제 비즈니스 도메인에서 이해하기 쉽고 설명 가능한 **투명한 통계 모델**을 지향합니다.
1. **평탄화된 예측 (Smoothing):** 최근의 출고량 추이를 기준으로 삼아 일시적 변동성(스파이크)을 제거한 안정적인 일/주간 예상 소진율을 도출합니다.
2. **원가 기반 자산 평가:** 단순히 마스터 정보의 예상 단가가 아닌, 실제 입고될 때 치렀던 인보이스의 **실제 결제 원화 금액**을 역추적하여 정확한 재고 금액과 폐기 리스크 비용을 산출합니다.
3. **직관적인 피드백 루프:** 프론트엔드 UI를 통해 수량을 입력하는 즉시, 재고일수가 어떻게 변하는지 색상으로 맵핑하여 직관적인 피드백을 제공합니다.

## 2. 코어 아키텍처 및 기술 스택

본 프로젝트는 고비용의 클라우드 인프라나 무거운 프레임워크를 지양하고, 독립된 사내 PC에서 빠르고 가볍게 구동되는 방향을 채택합니다.

### 2.1. Standalone Local ERP
사내 보안과 비용 최적화를 위해 로컬 호스트 자원을 최대로 활용하는 Standalone 구조입니다. 외부 노출이 없으며, 언제든 스냅샷 백업으로 복원이 가능합니다.

### 2.2. 프론트엔드
- **HTML/CSS/JS만 사용** — 프레임워크(React, Vue 등) 사용 금지, 속도와 경량화 초점.
- **CSS:** `web/static/css/style.css` 단일 파일 (API 지향 설계와 뷰 분리).
- **JS:** `web/static/js/app.js` 공통 유틸 + 각 HTML 페이지 내 인라인 `<script>`.
- **차트:** Chart.js 4.x (CDN)

### 2.3. 백엔드 및 데이터베이스
- **Python 3.11 + FastAPI + Uvicorn** — 프론트엔드와 철저한 API 분리.
- **데이터베이스:** SQLite 3 (`data/local_erp.db`) — 구성 간편, 백업 용이.
- **제약:** 머신러닝 라이브러리 사용 금지 (사칙연산 기반 예측만 허용).

### 2.4. 파일 구조
```
web/
├── static/
│   ├── css/style.css    ← 전역 CSS (유일한 스타일시트)
│   ├── js/app.js        ← 공통 JS (API, Format, Theme, Toast, ChartDefaults 등)
│   └── img/             ← 이미지 자산
├── index.html           ← 요약 대시보드
├── inventory.html       ← 현재고 현황
├── order_plan.html      ← 발주 계획
├── matching.html        ← 입고 파이프라인 매칭
├── expiry.html          ← 유통기한 방어 전략
├── users.html           ← 시스템 운영 및 마스터
└── login.html           ← 로그인
```

## 3. 핵심 기능별 기획 의도 및 비즈니스 로직

### 3.1. 전사 요약 대시보드 (Dashboard / index.html)
- **기획 의도:** 아침 출근 시 시급하게 조치해야 할 위기 상황(품절, 폐기 임박, 미매칭 건 등)을 한눈에 파악.
- **주요 기능:** 전사 합산 KPI 카드, 재고 추세 차트, 할일(To-Do) 알림 제공.

### 3.2. 현재고 현황 및 이관 (Inventory / inventory.html)
- **기획 의도:** 단순히 재고 파악을 넘어 채널별(온라인, 오프라인 등) 재고 배분을 최적화 및 이관 시뮬레이션.
- **주요 기능:** 거점별 실재고 및 자산 금액 조회, 이관 시뮬레이터 엑셀 업로드 지원.
- **이관 규칙:** 용인 메인창고(HUB) → 각 풀필먼트 창고로 이동 (한 창고에 여러 SKU 이동 가능).

### 3.3. 유통기한 방어 전략 (Expiry / expiry.html)
- **기획 의도:** 유통기한 임박 상품이 폐기되기 전 선제적 조치 알람 및 매몰 예상 비용 시각화.
- **주요 기능:** FEFO(선입선출) 기반 관리, 채널별 소진예상일 산출 및 예상 폐기 금액 추적.

### 3.4. 데이터 기반 발주 계획 (Order Plan / order_plan.html)
- **기획 의도:** 감이 아닌 출고량 통계 기반 제안 수량 도출 및 시각적 피드백 제공.
- **발주 규칙:** 리드타임 고려하여 **6개월 뒤 도착분을 주문**하는 것 (UI에 명확히 표시).
- **시뮬레이션:** 수량/가중치 변경 시 즉시 재계산 및 히트맵/차트 자동 반응. 상태는 수정 중(draft) ↔ 확정(confirmed).

### 3.5. 입고 파이프라인 매칭 (Inbound Matching / matching.html)
- **기획 의도:** 발주(당사) - 생산(제조사) - 인보이스(물류사) 내역의 교차 검증 및 누락 확인.
- **주요 기능:** 3단계 엑셀 업로드, 고유 식별자 기반 자동 매칭, 불일치 데이터 수동 매칭.

### 3.6. 마스터 및 시스템 운영 (System Admin / users.html)
- **기획 의도:** 로컬 시스템 안정성 보장을 위한 DB 백업 관리.
- **주요 기능:** 백업 스냅샷 생성 및 롤백 기능 지원.

### 3.7. 공통 기준 데이터 (Master Data)
- **상품 마스터:** (SN-001: 산양분유 1단계, SN-002: 산양분유 2단계, SN-003: 산양분유 3단계, SN-004: 산양분유 4단계).
- **창고 마스터:** (hub: 용인 메인, online: 온라인 FFC, offline: 오프라인 FFC, buyout: 바이아웃, coupang: 쿠팡 FFC).
- **재고일수 히트맵 기준 (3개월 = 13주 적정):**
  - `< 6주`: 빨강 (`risk-high`) / 위험
  - `6~9주`: 노랑 (`risk-mid`) / 주의
  - `9~13주`: 녹색 (`risk-low`) / 양호(적정)
  - `> 13주`: 파랑 (`risk-safe`) / 과잉

## 4. UI/UX 디자인 가이드라인

### 4.1. 브랜드 톤앤매너 및 색상
- **모던 B2B 룩앤필:** 장식적 애니메이션 지양, 깔끔한 폰트와 절제된 여백을 통한 데이터 집중.
- **색상 시스템:**
  - 메인 브랜드/긍정: `--accent-main` / `--accent-green` (#29AD3A)
  - 경고/주의: `--accent-amber` (#e08a00)
  - 위험/부정: `--accent-red` (#e53535)
- **아이콘:** 이모지 사용 금지, SVG 아이콘만 허용.
- **버튼 디자인:** 플랫 디자인 유지 (그라데이션 및 그림자 효과 금지).

### 4.2. 레이아웃 및 뷰
- **기본 프레임:** 좌측 사이드바(`app-sidebar`) + 우측 메인(`main-content`).
- **헤더:** 제목/소제목 중앙 정렬 (내부 `page-header-left` 래핑), 날짜 우측 상단 배치.
- **서브 네비게이션:** 콘텐츠 분리 필요 시 `sub-nav` + `sub-view` 탭 활용.
- **접기/열기(Collapsible):** 큰 영역은 `collapsible-header` + `collapsible-body` 및 `toggleCollapsible()` 적용.
- **데이터 방어:** 결측값(null) 발생 시 화면 깨짐을 막기 위해 `-`나 `소진 불가` 등 명확한 대체 텍스트 표시.

### 4.3. 날짜 및 테이블 포맷팅
- **주차 표기:** 영문 월약어 3글자 패턴 (예: `Jun-W3`). 한글 주차 표기 금지.
- **날짜 표기:** `2026년 6월 19일 (목)` 형식.
- **테이블 정렬:** 텍스트 좌측, 숫자 우측(`text-right`), 상태 중앙 정렬. 세로 구분선 활용. 고정폭 코드 폰트는 `mono` 클래스 사용.

## 5. 전체 개발 수정 원칙 (★ 최우선)

### 5.1. 연쇄 오류 방지 — "전수 검사 필수"
- **하나의 함수/변수/CSS 클래스를 수정하면, 해당 심볼을 참조하는 모든 파일을 반드시 검색하여 영향도를 확인한다.**
- 공통 유틸 (`app.js`), 전역 스타일 (`style.css`) 변경 시 모든 6개 HTML을 교차 검증해야 한다.
- HTML 수정 완료 후 DOM 태그 열기/닫기 매칭 및 요소 ID 중복 여부를 꼼꼼히 점검한다.

### 5.2. 샘플 데이터 무결성 및 Null Safety
- API 연결 여부와 관계없이 UI는 샘플 데이터만으로 완전히 동작해야 한다.
- 모든 API 응답 처리 시 `null` / `undefined` 가드를 둔다 (옵셔널 필드 fallback 처리 필수).
- **데이터 시딩 (Data Seeding):** 여러 PC에서 협업(GitHub)할 때 로컬 테스트가 용이하도록 `seed_data.py`를 적극 활용한다. 로컬 DB 초기화 및 UI 테스트 시 해당 스크립트를 실행해 파이프라인 전체가 연결된 풍부한 샘플 데이터를 구축한 후 검증한다.

### 5.3. app.js 공통 객체 및 함수 적극 활용
- **객체:** `API`, `Auth`, `Format`, `Theme`, `Sidebar`, `ChartDefaults`, `Toast`.
- **포맷팅:** `Format.number()`, `Format.currency()`, `Format.weekLabel()`, `Format.today()`.
- **기능:** `switchTab()`, `downloadTemplate()`, `toggleCollapsible()`, `injectTodayDate()`.

## 7. 주요 에러 해결 및 개선 이력 (Error Solving & Refactoring History)

### 7.1. API 무응답 (Hanging) 오류
- **증상:** API 요청 시 5번 이상 시도해도 아예 무반응 상태로 로딩만 지속되며 오류 로그도 없음.
- **원인:** 이전에 실행된 (혹은 비정상 종료된) Uvicorn 서버의 좀비 프로세스가 포트를 점유하여 발생한 데드락.
- **AI 행동 지침:** API 무응답 시 코드 내부 로직을 파헤치기 전에 **가장 먼저 서버 프로세스를 완전히 종료하고 재시작**하여 포트 충돌 여부를 확인할 것.

### 7.2. 전역 폰트 불일치 및 테이블 레이아웃 깨짐 현상
- **증상:** 폼 요소/테이블이 브라우저 기본 폰트로 렌더링되며, 데이터 문자열이 길어지면 테이블 UI가 붕괴됨.
- **원인:** CSS에서 폰트 상속 누락 및 `td`의 텍스트 오버플로우 방어 로직 부재.
- **AI 행동 지침:** 새로운 컴포넌트/테이블 추가 시 반드시 `max-width`, `overflow: hidden`, `text-overflow: ellipsis` 처리를 고려하고 폼 요소 폰트는 전역 변수를 따르도록 할 것.

### 7.3. 비효율적인 UI/UX 로직 (수동 시뮬레이션)
- **증상:** 서버 부하가 없는 프론트엔드 단순 연산임에도 "시뮬레이션 실행" 버튼을 일일이 눌러야 함.
- **AI 행동 지침:** 클라이언트 사이드에서만 이루어지는 가벼운 계산이나 시뮬레이션은 **별도 버튼 없이 실시간(Reactive)으로 즉각 반영**되도록 설계할 것.

### 7.4. Git 버전 관리 동기화 누락
- **증상:** AI가 로컬 코드만 수정하고 Github 원격 저장소에는 반영하지 않음.
- **AI 행동 지침:** 중요한 로직 수정이나 리팩토링 후에는 반드시 사용자의 동의를 구하고 `git add`, `git commit`, `git push`를 통해 코드를 원격 저장소와 동기화할 것.

### 7.5. Uvicorn 무응답 (Hanging) 및 프로세스 데드락 추가 해결
- **증상:** Uvicorn 서버 종료 시 APScheduler가 메인 스레드를 블로킹하여 좀비 프로세스화 됨.
- **원인 및 조치:** `scheduler.shutdown(wait=False)`로 변경하여 즉시 종료 허용.

### 7.6. 파이프라인 및 시스템 모듈 변수 참조 에러 (NameError / AttributeError)
- **증상:** `VALID_INBOUND_STATUSES` 미정의로 인한 크래시, `snapshot.file_path` 오참조로 인한 에러 발생.
- **원인 및 조치:** DB 모델 제약 조건에 맞춘 상수 선언 및 올바른 객체 속성(`snapshot_path`) 참조로 수정.

### 7.7. 시뮬레이션 로직 TypeError 방어 (Null Safety)
- **증상:** 발주 시뮬레이션 중 `pack_qty_per_tu` 필드가 `None`일 때 TypeError 발생.
- **원인 및 조치:** `if (product.pack_qty_per_tu and product.pack_qty_per_tu > 0)`로 Null 가드 강화.
